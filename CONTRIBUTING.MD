# 📝 팀 협업 가이드 (CONTRIBUTING.md)

우리 팀은 **정석적이고 안전한 개발 환경**을 지향합니다. 모든 팀원은 프로젝트 참여 전 아래 가이드를 반드시 숙지하고 준수해 주세요.

---

## 1. 이슈 및 프로젝트 보드 관리

모든 작업의 시작은 **이슈(Issue)** 등록입니다.

- **이슈 생성**: GitHub Issues에서 작업 성격에 맞는 템플릿(할 일/버그/도움)을 선택해 작성합니다.
- **보드 자동화**: 이슈가 생성되면 프로젝트 보드의 `Todo` 컬럼에 자동으로 등록됩니다.
- **번호 확인**: 생성된 이슈 번호(예: `#20`)는 브랜치 명과 커밋 메시지에 사용됩니다.

## 2. 브랜치 전략 (Git-flow 경량화)

- **`main`**: 상용 배포 브랜치입니다. 직접 `push`가 금지되며 보호 규칙이 적용됩니다.
- **`develop`**: 개발 기준 브랜치입니다. 모든 기능 브랜치의 출발점이자 합류점입니다.
- **`feature/이슈번호-기능명`**: 개별 작업 브랜치입니다. (예: `feature/20-button`)

### 작업 시작 프로세스

```bash
git checkout develop
git pull origin develop
git checkout -b feature/이슈번호-작업명
```

## 3. 커밋 메시지 규칙 (Conventional Commits)

`commitlint`와 `husky`가 설정되어 있어 규칙 위반 시 커밋이 차단됩니다.

- **형식**: `type: 메시지 (#이슈번호)`
- **주요 타입**:
  - `feat`: 새로운 기능 추가
  - `fix`: 버그 수정
  - `docs`: README 등 문서 수정
  - `style`: 코드 포맷팅 (로직 변경 없음)
  - `refactor`: 코드 리팩토링
- **예시**: `feat: 프로젝트 초기 디렉토리 구조 설정 (#20)`

## 4. 코드 품질 관리

- **자동 포맷팅**: VS Code에서 파일 저장 시 ESLint와 Prettier가 코드를 자동 정리합니다.
- **커밋 전 검사**: 커밋 시점에 `lint-staged`가 실행되어 코드 결함을 체크합니다. 에러 발생 시 수정한 후에만 커밋이 완료됩니다.

## 5. PR(Pull Request) 및 코드 리뷰

1. 작업 완료 후 `develop` 브랜치 방향으로 PR을 생성합니다.
2. PR 본문에 **`Closes #이슈번호`**를 포함하여 머지 시 이슈가 자동 종료되게 합니다.
3. **최소 1명 이상의 승인(Approve)**을 받아야 머지가 가능합니다.
4. 모든 리뷰 의견(Conversation)이 해결(Resolved)된 후 머지 작업을 진행합니다.
5. **`Squash and merge`**를 사용하여 커밋 히스토리를 한 줄로 깔끔하게 유지합니다.

## 6. 알림 및 소통

- **실시간 공유**: GitHub의 모든 활동(이슈, PR, 푸시)은 설정된 **Discord 채널**로 실시간 공유됩니다.
- **도움 요청**: 작업 중 막히는 부분이 있다면 `도움 요청` 라벨을 달아 이슈를 발행하거나 디스코드에서 팀원들에게 공유해 주세요.
